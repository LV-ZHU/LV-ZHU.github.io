从 0 到 1 完成 bot 版本的 Hello World

本地部署较为容易，可参考 LLBot 安装指南 和 LLbot 对接 AstrBot ，安装后启动llbot.exe、登录 QQ 号开始监听消息，再启动 AstrBot 的 start.bat 按终端提示访问 WebUI 即可。下面主要讲讲服务器部署。

服务器选型与环境搭建

阿里云新账号有300元云服务器ECS免费试用，默认是 Windows Server，但由于 Windows 系统大量资源会消耗在图形化上，且诸如宝塔、1Panel等服务器管理软件普遍对 Linux 系统性更好，于是更换为 Ubuntu 镜像。连接服务器时尽量采用Xshell连接，若使用阿里云自带连接方式尽量不使用VNC连接实例，VNC对扩展 ASCII 码适配较差，命令行中文会乱码，且会出现英文字母大小写问题。尽量使用Workbench或SSH连接。另外注意服务器需要按需放行诸如8888、6199、6185、6090、3080等端口。

Docker 装 AstrBot

宝塔 Docker 、端口映射

宝塔面板使服务器更便于管理（本质上是给部分命令行操作提供了WebUI界面），参考 宝塔面板容器部署astrbot 完成操作，一定要注意安装好astrbot容器后进入容器编排在配置文件里添加- ${QQ_WS_PORT}:6199 并在对应.env中添加环境变量 QQ_WS_PORT=6199，从而使 Napcat 和 Astrbot 的内网通信端口正常工作，也可以用下面的命令行方式:

docker run -d --restart always \ -p 6185:6185 -p 6199:6199 \ -v ./data:/AstrBot/data \ --name astrbot \ soulter/astrbot:latest

连通钉钉时遇到的问题及解决方案

装好了 AstrBot 就可以接入大多数消息平台了，我遇到的问题是钉钉 Stream 模式接入时反复报 401 authFailed。

日志现象

AstrBot 启动后，日志里第一次连接成功获取了 endpoint：

endpoint is {'endpoint': 'wss://wss-open-connection-union.dingtalk.com:443/connect', ...}

但紧接着几秒后第二次连接就报错了：

open connection failed, error=401 Client Error: Unauthorized ... response.text={"code":"authFailed","message":"鉴权失败"}

明明 endpoint 已经拿到了，为什么第二次连接会被拒？

排查过程

1.排除凭证问题。在终端用 curl 手动测试 AppKey 和 AppSecret：

curl -X POST "https://api.dingtalk.com/v1.0/oauth2/accessToken" -H "Content-Type: application/json" -d '{"appKey":"...","appSecret":"..."}'

返回了 accessToken，说明凭证没问题。

2.排除 IP 白名单。确认服务器出口 IP 与钉钉开发平台上配置的一致：

curl ifconfig.me

返回的 IP 与后台白名单一致，排除。

3.检查系统时钟。钉钉的鉴权机制基于 Token 签名，对服务器时间非常敏感。如果系统时钟与标准时间偏差过大，生成的签名会直接失效：

date ntpdate ntp.aliyun.com

检查后发现时钟偏差只有 -0.000598 秒，排除。

4.检查发布状态。在钉钉开放平台，改了 IP 白名单、重置 Secret 或更改权限后，必须去「版本管理与发布」点「确认发布」，不然线上网关跑的还是旧配置。钉钉的 API 校验（curl 拿 token）和长连接网关（Stream Connection）是不同的逻辑，curl 拿 Token 只要 AppKey/Secret 对了就能给，但 connections/open 需要连接网关确认你的应用当前线上版本是否开启了 Stream 能力。发现顶部有黄色警告条"版本发布后，当前修改才能生效"，点击发布。但发布后依然 401。

5.发现问题：多实例并发。看进程列表：

ps -ef | grep python

发现同一个 main.py 有多个 PID 在跑。原因是宝塔面板的「Python 项目管理器」默认开了异常重启——手动 kill 一个进程，它秒级拉起一个新的。新旧连接在钉钉网关处打架，而钉钉 Stream 模式一个 AppKey 只允许一个并发连接，多出来的全部返回 401。

解决方案

先在宝塔面板停止项目、关闭异常重启，然后清掉所有残留进程，等几分钟让钉钉网关释放旧连接，最后手动单次启动成功：

pkill -9 -f main.py ps -ef | grep python

安装 NapCat

Shell 模式 vs Docker 模式

NapCat 有多种 部署方式 ，执行一键安装命令后还会提供两种模式选择：Docker 模式（镜像自带完整环境，资源隔离但稍重）和 Shell 模式（直接跑在宿主机上，轻量但依赖系统库）。选了 Shell 模式，官方安装脚本会交互式询问协议类型，注意输入协议名（比如此处就是反向WS reverse_ws），另外不建议装tui，命令行环境下操作可视化界面并不友好效果不如直接输入命令。

Shell 模式还有一些系统依赖要装，按照提示操作，阿里云精简版 Ubuntu 默认不带这些图形渲染库，不装的话 QQ 核心进程无法启动（报 GPU 初始化错误）：

apt-get install -y xvfb libnss3 libasound2 libgbm1 ttf-wqy-zenhei

连通 QQ 时遇到的问题及解决方案

NapCat 和 AstrBot 都装好了，但消息就是不通。这个问题的原因可能出在链路的任何一环，需要逐一排查，一定要结合日志具体分析。

排查过程

1.确认端口是否真正监听。在宿主机上测试端口连通性，telnet 127.0.0.1 6199 显示 Connected，但 NapCat 日志里永远是「未启动」。宿主机的 docker-proxy 就像个前台，即便后面容器里没人也会先把电话接起来，但数据转发到容器内部后石沉大海。钻进容器里验证：

docker exec -it astrbot netstat -tulnp | grep 6199

返回为空，实锤：AstrBot 容器内部的 OneBot 服务根本没开。

2.检查连接模式——HTTP vs WebSocket。去 AstrBot 网页端检查 OneBot11 的连接模式，如果被设成了「正向 HTTP」而不是「反向 WebSocket」，AstrBot 在 6199 端口启动的就是 HTTP 服务，而 NapCat 发起的是 WebSocket 握手请求，协议不匹配，连接自然被拒绝。确认 AstrBot 侧设为「反向 WebSocket 服务端」（监听模式），NapCat 保持客户端去连它。

3.检查 WebSocket 路径是否一致。NapCat 配置里写的 ws://172.17.0.1:6199/ws，但 AstrBot 那边的路径可能没配 /ws 后缀。URL 路径必须严格一致，否则握手阶段直接 404。最终两边都改成根路径 ws://172.17.0.1:6199/（去掉 /ws），连接才通。还有一个附带的坑：AstrBot 在 Docker 容器内不能用 127.0.0.1 访问宿主机上的 NapCat，得用 Docker 网桥地址 172.17.0.1（127.0.0.1 在容器眼里是容器自己，不是宿主机）。

4.检查 Linux QQ 是否已登录。以上三层都排除后，NapCat 日志里依然是「未启动」。直到看到这一行：快速登录错误：你的用户身份已失效，请重新登录。在 QQ 扫码成功之前，NapCat 的 WebSocket 服务不会启动，它不会报错，就安静地显示「未启动」，让你以为是配置问题——QQ 没登录，后面的链路全都不通。扫码登录后，日志里瞬间出现 [info] 登录成功 和 [info] [OneBot11] [WS] 反向 WebSocket 连接成功。教训：先扫码登录，再排查配置。顺序反了就是在浪费时间。

排查技巧

NapCat 的详细日志不在屏幕输出里，真正的错误藏在 /root/.config/QQ/NapCat/logs/ 目录下的日志文件中，用 tail -f 实时追踪才能看到 WebSocket 握手失败的具体原因（403 Token 错、404 路径错、ECONNREFUSED 端口没开）。最终利用安装 NapCat 时提供的 token 登录 6099 WebUI 界面成功连通。

运维与稳定性

systemd 守护进程

SSH 一关机器人就掉线，服务器重启更是一切归零，nohup 和 screen 能解决 SSH 断开的问题（screen -dmS napcat 创建后台会话，Ctrl+A 再按 D 退出但不关闭），但服务器重启后它们无法自动恢复——最靠谱的方案是 systemd。

创建服务文件 /etc/systemd/system/napcat.service：

[Unit] Description=NapCat QQ Bot Service After=network.target [Service] WorkingDirectory=/root/Napcat/opt/QQ/ ExecStart=/usr/bin/xvfb-run -a /root/Napcat/opt/QQ/qq --no-sandbox -q 2913425420 Restart=always RestartSec=5 StandardOutput=syslog StandardError=syslog SyslogIdentifier=napcat User=root [Install] WantedBy=multi-user.target

写这个文件的时候踩了好几个坑。一开始用 nano 编辑，结果把 systemctl daemon-reload 这类终端命令也一起写进了 .service 文件里，导致 systemd 解析失败。后来 nano 异常退出，留下了 .napcat.service.swp 交换文件，再打开时内容全乱了。最终用宝塔面板的文件管理器直接新建文件、粘贴内容、保存，才搞定。

还有一个坑：xvfb-run 的路径问题。service 文件里写的 /usr/bin/xvfb-run，但有些系统装完 xvfb 后可执行文件不在 /usr/bin 下，用 which xvfb-run 确认实际路径，否则 systemd 启动时会报 No such file or directory。

三连命令激活：

systemctl daemon-reload systemctl enable napcat systemctl start napcat

验证状态：

systemctl status napcat

看到绿色的 active (running) 就稳了。之后看日志用 journalctl -u napcat -f，实时滚动输出。

扫码登录与 Token 获取

挂成守护进程后，终端里看不到二维码了。三条路找回来：

1. journalctl 挖日志：journalctl -u napcat -f，实时滚动输出，二维码和 Token 都在里面。

2. 直接看配置文件：cat /root/Napcat/.../config/webui.json，找到 "token": "xxxx"。

3. 6099 WebUI（最优雅）：浏览器访问 http://公网IP:6099/webui，输入 Token 登录后，QQ 登录状态、掉线重连一目了然。

QQ 反复掉线的排查

systemd 配好后，QQ 偶尔会莫名掉线然后一直卡在「未启动」。排查后发现是重启 NapCat 服务时旧进程没杀干净——QQ 核心进程（qq）、NapCat 进程（napcat）和虚拟显示进程（Xvfb）三个可能残留。重新启动前必须先清场：

pkill -9 qq pkill -9 napcat pkill -9 Xvfb

清完再 systemctl start napcat，否则新旧进程抢端口，QQ 会反复登录又掉线。

容器内无编辑器自救

进 AstrBot 容器改配置文件，nano/vi/vim 全 command not found

精简 Docker 镜像就是这么狠，连个编辑器都不装，终极方案是用 Here Document 直接覆盖文件：

cat < cmd_config.json { "dashboard": { "enable": true, "username": "linux", "password": "f086c50e945a45e7f903458c5d491911", "jwt_secret": "ac666104ba789748e1bccd2d53e06e269331d8b8076d14310ca97a42508d7140", "host": "0.0.0.0", "port": 6185 } } EOF

注意 password 字段必须是 32 位小写 MD5。改完后 exit 退出容器，在宝塔面板重启容器才生效。

插件依赖补装

插件报依赖缺失？依赖必须装在容器里面，不是宿主机

docker exec -it astrbot /bin/bash cd /AstrBot/data/plugins/[插件目录] pip install --root-user-action=ignore -r requirements.txt playwright install --with-deps chromium

Web 分析类插件需要额外装 chromium 浏览器内核，不然会报错。装完重启容器。

服务器重启后的恢复

重启后所有面板和 Bot 都打不开？按顺序排查

1. Docker 容器是否自启：检查是否设置了 --restart always。没有的话去宝塔 Docker 界面手动启动。

2. 宝塔面板入口：SSH 登录后执行 bt 14 查看最新入口地址和端口。

3. 安全组是否被重置：去阿里云控制台确认 6185、6099、6199 等端口是否仍然放行。

4. NapCat 状态：systemctl restart napcat，然后 systemctl status napcat 确认是否恢复。

docker ps -a bt 14 systemctl restart napcat systemctl restart bt

结语

实际写代码的时间远少于看日志、查文档、和 AI 排错的时间。大部分 debug 卡在「配置看着没问题但就是不通」的阶段，最后发现要么是少点了一个「发布」按钮（钉钉），要么是 Docker 网桥地址写错了（127.0.0.1 vs 172.17.0.1），要么是 QQ 没扫码导致后面的链路全部静默失败。
