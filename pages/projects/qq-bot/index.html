从 0 到 1 完成 bot 版本的 Hello World

本地部署较为容易，可参考 LLBot 安装指南 和 LLbot 对接 AstrBot ，安装后启动llbot.exe、登录 QQ 号开始监听消息，再启动 AstrBot 的 start.bat 按终端提示访问 WebUI 即可。下面主要讲讲服务器部署。

服务器选型与环境搭建

阿里云新账号有300元云服务器ECS免费试用，默认是 Windows Server，但由于 Windows 系统大量资源会消耗在图形化上，且诸如宝塔、1Panel等服务器管理软件普遍对 Linux 支持性更好，于是更换为 Ubuntu 镜像。连接服务器时尽量采用Xshell连接，若使用阿里云自带连接方式尽量不使用VNC连接实例，VNC对扩展 ASCII 码适配较差，命令行中文会乱码，且会出现英文字母大小写问题。尽量使用Workbench或SSH连接。另外注意服务器需要按需放行诸如8888、6199、6185、6090、3080等端口。

Docker 装 AstrBot

宝塔 Docker 、端口映射

宝塔面板使服务器更便于管理（本质上是给部分命令行操作提供了WebUI界面），参考 宝塔面板容器部署astrbot 完成操作，一定要注意安装好astrbot容器后进入容器编排在配置文件里添加- ${QQ_WS_PORT}:6199 并在对应.env中添加环境变量 QQ_WS_PORT=6199，从而使 Napcat 和 Astrbot 的内网通信端口正常工作，也可以用下面的命令行方式:

docker run -d --restart always \ -p 6185:6185 -p 6199:6199 \ -v ./data:/AstrBot/data \ --name astrbot \ soulter/astrbot:latest

连通钉钉时遇到的问题及解决方案

装好了 AstrBot 就可以接入大多数消息平台了，我遇到的问题是钉钉 Stream 模式接入时反复报 401 authFailed

日志现象

AstrBot 启动后，日志里第一次连接成功获取了 endpoint：

endpoint is {'endpoint': 'wss://wss-open-connection-union.dingtalk.com:443/connect', ...}

但紧接着几秒后第二次连接就报错了：

open connection failed, error=401 Client Error: Unauthorized ... response.text={"code":"authFailed","message":"鉴权失败"}

排查过程

1.排除凭证问题。在终端用 curl 测试 AppKey 和 AppSecret：

curl -X POST "https://api.dingtalk.com/v1.0/oauth2/accessToken" -H "Content-Type: application/json" -d '{"appKey":"...","appSecret":"..."}'

返回了 accessToken，说明凭证没问题。

2.排除 IP 白名单。确认服务器出口 IP 与钉钉开发平台上配置的一致：

curl ifconfig.me

返回的 IP 与后台白名单一致，排除。

3.检查系统时钟。钉钉的鉴权机制基于 Token 签名，对服务器时间非常敏感。如果系统时钟与标准时间偏差过大，生成的签名会直接失效：

ntpdate ntp.aliyun.com

检查后发现时钟偏差只有 -0.000598 秒，排除。

4.检查发布状态。在钉钉开放平台，改了 IP 白名单、重置 Secret 或更改权限后，必须去版本管理与发布点确认发布，不然线上网关跑的还是旧配置。钉钉的 API 校验（curl 拿 token）和长连接网关（Stream Connection）是不同的逻辑，curl 拿 Token 只要 AppKey/Secret 对了就能给，但 connections/open 需要连接网关确认你的应用当前线上版本是否开启了 Stream 能力。发现顶部有黄色警告条"版本发布后，当前修改才能生效"，点击发布。但发布后依然 401。

5.发现问题：多实例并发。看进程列表：

ps -ef | grep python

发现同一个 main.py 有多个 PID 在跑。原因是宝塔面板的Python 项目管理器默认开了异常重启，只要进程被 kill 就会立刻拉起一个新的，这样钉钉网关就会同时收到多份连接。而钉钉 Stream 模式一个 AppKey 只允许一个并发连接，多出来的全部返回 401。

解决方案

先在宝塔面板停止项目、关闭异常重启，然后清掉所有残留进程，等几分钟让钉钉网关释放旧连接，最后手动单次启动成功：

pkill -9 -f main.py ps -ef | grep python

安装 NapCat

Shell 模式 vs Docker 模式

NapCat 有多种 部署方式 ，执行一键安装命令后还会提供两种模式选择：Docker 模式（镜像自带完整环境，资源隔离但稍重）和 Shell 模式（直接跑在宿主机上，轻量但依赖系统库）。选了 Shell 模式，官方安装脚本会交互式询问协议类型，注意输入协议名（比如此处就是反向WS reverse_ws），另外不建议装tui，命令行环境下操作可视化界面并不友好效果不如直接输入命令。

Shell 模式还有一些系统依赖要装，按照提示操作，阿里云精简版 Ubuntu 默认不带这些图形渲染库，不装的话 QQ 核心进程无法启动（报 GPU 初始化错误）：

apt-get install -y xvfb libnss3 libasound2 libgbm1 ttf-wqy-zenhei

连通 QQ 时遇到的问题及解决方案

常见现象是 NapCat 和 AstrBot 都装好了，但消息就是不通。可能的原因很多，需要结合日志一个个排查。

配置文件方面，反向 WebSocket 一定要配对。NapCat 的配置文件路径在 root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config 下，包括连接类型/ws、Napcat WebUI的 token 都在该目录下。

端口连接方面，先在安全组放行端口。在容器里运行 telnet 127.0.0.1 6199，看到 Connected 就说明通了；然后可以再模拟 astrbot 的连接行为：docker exec -it astrbot netstat -tulnp | grep 6199。

排查技巧

用安装 NapCat 时提供的 token 登录 WebUI（http://公网IP:6099/webui），可以直观看到 QQ 登录状态、服务端开启情况等信息。

从 1 到 10 确保 bot 稳定在线

开启守护进程

成功发出消息后最重要的需求就是确保 Astrbot+Napcat 能稳定运行。nohup 和 screen 指令能解决连接断开的问题（screen -dmS napcat 创建后台会话，Ctrl+A 再按 D 退出但不关闭），但服务器重启后它们无法自动恢复。比较好的方案是 systemd 守护进程。

创建服务文件在 /etc/systemd/system/napcat.service 目录下，具体的诸如 xvfb-run 的路径可以用 where 指令查询后替换：

[Unit] Description=NapCat QQ Bot Service After=network.target [Service] WorkingDirectory=/root/Napcat/opt/QQ/ ExecStart=/usr/bin/xvfb-run -a /root/Napcat/opt/QQ/qq --no-sandbox -q <此处替换为机器人要登录的 QQ 号> Restart=always RestartSec=5 StandardOutput=syslog StandardError=syslog SyslogIdentifier=napcat User=root [Install] WantedBy=multi-user.target

输入命令激活：

systemctl daemon-reload systemctl enable napcat systemctl start napcat

验证状态：

systemctl status napcat

看到绿色的 active (running) 就说明守护进程已经开好了。以后确认 bot 工作状态是否正常可以在 QQ 聊天框里发 #Napcat，如果没有响应，可以用 journalctl -u napcat -f看日志，或者直接登录http://IP地址:6099/webui看 QQ 登录情况。如果连各类 WebUI 都登不上了，可能是服务器实例规格不够的原因，可检查服务器运行状态，升级服务器规格或者强制重启。

排查 bot 问题时如果需要修改配置文件可以在找到对应文件所在目录后 cat 和重定向命令覆盖。
