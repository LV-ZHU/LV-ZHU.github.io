一、概要

本地部署（Windows）

  • Astrbot 在本地运行，使用 start.bat 一键启动。
  • LLbot 使用 LLbot Desktop，WebUI 端口为 3080。
  • 作用：先验证插件与命令，再同步到服务器。

服务器部署（Linux）

  • Astrbot 运行在 Docker 容器内。
  • NapCat 运行在宿主机上，负责 QQ 协议链路。
  • 作用：无人值守、重启可恢复、长期在线。

消息平台

  • 钉钉
  • 飞书
  • QQ
  • 个人微信

说明

  • 新人优惠存在，但活动经常变化，以官网实时页面为准。
  • 文中公网地址统一脱敏为 [公网IP]。
  • 端口示例：6185、6199、8888、6099、3080。

三、具体流程与问题展开（服务器为主）

步骤1：服务器选型与系统迁移（Windows -> Ubuntu）

现象： 默认系统为 Windows Server，部署容器和依赖时效率低。

处理： 直接用镜像重装 Linux，后续统一在 SSH 下操作；VNC 仅作为应急入口，不作为常规方式。

补充： 新人优惠有活动窗口，具体套餐按阿里云实时页面为准。

ssh-keygen -t ed25519 ssh-copy-id [user]@[公网IP] ssh [user]@[公网IP]

步骤2：宝塔面板安装与容器入口建立

现象： Windows 版宝塔可用信息少，问题定位成本高。

处理： 服务器统一使用 Linux 版宝塔，依托 Docker 管理 Astrbot 容器。

docker ps docker logs [容器名] docker logs -f [容器名]

步骤3：实例重启后的监听恢复（NapCat TUI 先打通）

现象： 重启实例后，协议端恢复慢，常误判为配置文件损坏。

处理： 先跑 NapCat TUI 监听，确认协议链路正常后再回到 WebUI 和 Astrbot 对接。

./napcat.sh -q [QQ号] journalctl -u napcat -f

步骤4：Docker 环境变量与 6199/6200 端口冲突

现象： Astrbot 与 NapCat 连接失败，6199 不可达或经常冲突。

原因： 端口映射、容器监听地址、安全组三方不一致；有时只是 docker-proxy 有响应，容器服务本身未监听。

处理： 确认容器内监听到 0.0.0.0；必要时将通信端口迁移到 6200，并同步修改双方配置和安全组。

docker exec -it [容器ID] netstat -tulnp netstat -tunlp | grep 6199 netstat -tunlp | grep 6200

步骤5：NapCat WebUI 状态监控（6099）

现象： 机器人偶发离线，但终端日志不一定直接提示根因。

处理： 通过 6099 WebUI 查看登录态、掉线重连、二维码与 Token 状态。

netstat -tunlp | grep 6099 journalctl -u napcat -f

步骤6：Linux 自启动守护（systemd）

现象： SSH 断开或服务器重启后，NapCat 进程退出。

处理： 使用 systemd，设置 Restart=always 与开机自启。

systemctl daemon-reload systemctl enable napcat systemctl start napcat systemctl status napcat
[Unit] Description=NapCat QQ Bot Service After=network.target [Service] WorkingDirectory=/root/Napcat/opt/QQ/ ExecStart=/usr/bin/xvfb-run -a /root/Napcat/opt/QQ/qq --no-sandbox Restart=always RestartSec=5 StandardOutput=syslog StandardError=syslog SyslogIdentifier=napcat User=root [Install] WantedBy=multi-user.target
扩展难点A：后台运行后看不到二维码或 Token

处理： 用 journalctl 追踪服务输出；同时检查 NapCat 配置目录中的 WebUI 配置。

journalctl -u napcat -f cat /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/webui.json
扩展难点B：容器里没有 vi/nano，配置改不动

处理： 使用 Here Document 直接写文件；Astrbot 密码字段使用 32 位小写 MD5，修改后重启容器。

cat <<EOF > cmd_config.json docker restart [容器名]
扩展难点C：插件依赖缺失（例如 Playwright）

处理： 依赖必须安装在容器内部，不是宿主机。

docker exec -it [容器名] /bin/bash pip install -r requirements.txt playwright install --with-deps chromium
扩展难点D：OneBot 正向/反向角色混淆

处理： 建议 Astrbot 作为反向 WS 服务端，NapCat 作为客户端拨号端。Token 和路径必须一致；必要时先用根路径做最小验证。

tail -f /root/.config/QQ/NapCat/logs/[日志文件]
扩展难点E：服务器重启后面板和 Bot 全部失联

处理： 先检查容器重启策略，再查宝塔入口与关键服务状态。

docker ps -a bt 14 docker start [容器名]
扩展难点F：进程残留导致重复连接或端口占用

处理： 清理残留进程后再重启服务，避免旧连接干扰。

ps -ef | grep python pkill -9 qq pkill -9 napcat pkill -9 Xvfb
扩展难点G：文件权限、路径与编辑问题（容器内/宿主机）

处理： 先确认当前目录，再确认脚本可执行权限；需要替换配置时优先用可审计命令。

cd [路径] ls ls -F chmod +x [文件名] cat [文件名] nano [文件名] vi [文件名] sed -i 's/旧/新/g' [文件名]
扩展难点H：系统依赖、会话管理与滚动日志

处理： 先补系统依赖再启动协议端；调试期可用 screen 看运行态，正式环境使用 systemd。

sudo apt-get update sudo apt-get install -y [包名] screen -ls screen -r [会话名] screen -dmS [会话名] [命令] journalctl -u napcat -f
扩展难点I：Docker 容器生命周期管理

处理： 先确认现有容器状态，再决定重启、删除重建或新增运行。

docker ps docker logs [容器名] docker logs -f [容器名] docker rm -f [容器名] docker start [容器名] docker run [参数] [镜像名]
扩展难点J：NapCat 启动模式选择（图形模拟 vs Shell 模式）

处理： 服务器环境优先使用 Shell 模式；图形模拟仅在必要时使用，避免额外资源占用。

./napcat.sh -q [QQ号] xvfb-run -a [命令] node ./main.js -q [QQ号]
扩展难点K：钉钉 401 的细分排查（发布、IP、并发、时钟）

排查顺序： 先确认应用发布版本，再确认真实出口 IP 白名单，再检查是否有旧进程抢占连接，最后核对系统时间。

curl ifconfig.me timedatectl ps -ef | grep python kill -9 [PID]
